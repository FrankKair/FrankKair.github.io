This is the first post in the series **JSfy** where I will talk about my experience using JavaScript in the last years.

### Language and community

Once I was talking to a group of programmers and one of them was getting started with JavaScript, having previous experience with other languages. At first he said that the language was horrible, that it is too easy to make mistakes. Then, he said that the language was ok, but the community, the people were awful. Is there some truth to any of these comments? Is the language that bad or is the community around it that awful?

These statements made me realise that, at this point, JavaScript is so ubiquitous that everyone has an opinion about it. I even got asked about my opinion on the term "JavaScript fatigue" in an interview once (more on that later). Sometimes, people form their opinion after using the tools for a long time, having found the good, the bad and the ugly in the language. On other occasions, people read blog posts, or merely talk to other developers and create a misconception. 

It’s very easy to spread misconceptions, having no context whatsoever, and often times, these things stick around.


### Context

JavaScript’s origin story[^fn1] is quite interesting. In 1995, Brendan Eich, at the time working for Netscape Communications, was tasked to develop a scripting language for the browser. The new language was going to be based on Scheme[^fn2], but then something happened. Sun Microsystems was about to close a deal with Netscape, where they would include Java in the browser as well. 

In that scenario, Java is the big, enterprise language and they wanted a scripting language to be the lightweight counterpart. Available options at that time includes Perl, Python, TCL and Scheme, but none of them resembled Java in the slightest. Netscape wanted to make the case that their language was up for the task.

That dramatically changed the fate of JavaScript, a language that initially was supposed to be “Scheme for the browser” now inherits the syntax of Java, with curly braces instead of parenthesis. Semantically, though, JavaScript still maintained its functional roots. A prototype was created in 10 days in May 1995[^fn3].

It is easy to see how the language could have flaws, knowing that its first prototype was put together in 10 days, under a lot of pressure. It seems to me that the language might have been marketed as a less serious language, where Java got all the professional praise. It is also worth pointing out that during the period, the web was in its infancy, and JavaScript was not being used by professional engineers, but rather by anyone trailblazing the internet.

### Language

JavaScript is very permissive. That’s both a blessing and a curse. Being dynamic and weakly typed, you can create programs very quickly, run them and see things happening, even if there are small mistakes lurking. Here’s an example:

```javascript
const number = 50;
const text = "Hello";
console.log(number + text);
```

First of all, notice that the operator “+” is being used with two different types, string and number. The output is “50Hello”, which is odd to be allowed in the first place. But if you think about it, we just said that the language is *weakly* typed, so it should not be a surprise that there is automatic typecasting.

Novices will find joy creating programs, even with some minor mistakes. And that might be a good thing, that these first steps can be done more easily, with less associated cognitive load. Compare that with having, say, C, C++ or Java as your first language. It might be the case that if you’re not studying Computer Science, you just want to learn to build some cool things, and starting with a language that is “blocking” your progress may be just enough to make you give up.

As for more experienced engineers, I’m under the impression that very few actually learned JavaScript. Most people I’ve come across just started using it and took it from there. Again, it is a blessing and a curse that you can just start using the language and most things just work (it reminds me a little bit of Ruby in that regard). The curse aspect is that people may never really understand the language (prototype? this? bind?). Douglas Crockford called JavaScript “Lisp in C’s Clothing”[^fn4].

It became my personal quest to fully learn and understand the language, so I’ve read the following books:

- JavaScript: The Good Parts (Douglas Crockford);
- Effective JavaScript (David Herman);
- Eloquent JavaScript (Marijn Haverbeke);
- How JavaScript Works (Douglas Crockford).

(I will give an overview of these books in another post).

So… to answer the first question, do I think JavaScript is a bad language? No, I don’t think it is. In its current version, I find the language (and its ecosystem) very powerful and fun to use. I do, however, find it odd that, for example, *classes* were introduced in the language, since we already have a *prototype* system in place. I think that gives a shortcut to developers that are not willing to learn a different concept (namely prototyping and prototype chain). I'm a big believer that languages should have less facelifts and more consistency.

What Douglas Crockford wrote here[^fn5] pretty much sums it up:

- "JavaScript is an astonishing language, in the very worst sense. Its astonishing nature led to a very bad reputation";

- "But despite JavaScript's astonishing shortcomings, deep down, in its core, it got something very right. When you peel away the cruft, there is an expressive and powerful programming language there".


### Community

The second question: Is the community bad? For me that is very quick answer: No.

The community effectively "ported" JavaScript to the server ([Node.js](https://nodejs.org/en/)), the desktop ([Electron](https://electronjs.org)) and mobile ([React Native](https://facebook.github.io/react-native/)). Whether you think it's good to use JavaScript in these scenarios or not, that's another story, but we must be able to admit that it is amazing how much people embraced JavaScript and made it the most popular programming language ever[^fn6]. By the way, that's why the title of this series is called JSfy, because the community if JSfying the programming world.

### Fatigue vs renaissance

An interviewer asked me "What do you think about JavaScript fatigue?" and I also remember reading somewhere "We're not living a JavaScript fatigue, we're going through JavaScript renaissance!". You can make the case for both sides. Indeed, too many things are happening too fast in the JS community, and it's hard to keep up with everything. On the bright side, how awesome is the fact that there is so much activity all the time?

### So what?

I think we should spread less misconceptions. It's one thing if you have your opinion and want to discuss it, but bashing languages (or anything really), without proper context and understanding, just piping what one reads on the internet, it's just counter productive to all tech communities.
